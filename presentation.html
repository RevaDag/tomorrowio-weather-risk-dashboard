Part 01 — The Problem

No quick way to check all locations

Users with multiple saved locations open each one individually. There is no quick way to see which location needs attention today.

Alerts only appear for emergencies. The common case is missed: nothing is critical, but is anything worth checking?

Send too many alerts — and users ignore all of them. More notifications does not mean more awareness. Users stop paying attention, and the product loses value.

Part 01 — What We're Building

Three core behaviors

Automatic Summary

Shows a 7-day outlook across all saved locations. No questions needed.

Impact Ranking

Shows the top 2–3 locations with the highest risk. Users can ask for more.

Follow-up Q&A

Ask follow-up questions about a specific location, day, or condition.

How we rank locations

①

Active severe-weather alerts Events API

Highest weight
②

Weather values that pass a dangerous threshold

Temp extremes · heat stress index Heavy precip · snow · freezing rain · hail Flood index · fire index · lightning Wind — sustained + gusts Air quality · pollen Low visibility (fog) · extreme UV
Mid weight
③

Unusual weather vs. normal patterns

Unusually cold weather after a warm week ranks higher than a regular cold period.

Lower weight
All thresholds are adjustable after launch — no new release needed. We tune them based on how real users respond.

Part 02 — Prototype

From design scan to working mock

Step 1

Capture Tomorrow.io's design language

Used the Claude browser extension to scan the live Tomorrow.io web app — extracting its layout, color palette, typography, spacing, and component styles directly from the page. No design system doc needed.

Dark navy bg Muli + Roboto Mono Brand gradient Card & chip patterns Spacing tokens
Step 2

Build the mock with Claude Code

Fed the style guide into Claude Code using the frontend-design skill — which generates production-grade UI that matches the exact look and feel of the existing product.

100% static · no backend HTML / CSS / JS Deployed to Vercel
The result looks and feels native to Tomorrow.io — users recognise the product immediately. The whole prototype was built in a single afternoon.

Built for decisions, not data display

Risk scores are ranked by severity. Ask Gale to get dollar impact and staff estimates. Every screen ends with a recommendation, not raw numbers.

Gale talks business, not weather

"Reschedule Tuesday's outdoor crew" — not "60 mm of rain." Ready-made prompts make it easy to start.

The Thread

Both parts share one idea

🔀

Rank, don't list

Show users what matters most, not everything. Both the chat and the dashboard start with the highest-risk item first.

🧠

Interpret, don't deliver

Tomorrow.io's advantage is turning data into clear insight. Both features make that advantage stronger.

⚡

Recommend, don't observe

Operations managers need to know what to do next. Every screen ends with a clear, concrete action.
